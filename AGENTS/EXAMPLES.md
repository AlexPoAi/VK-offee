## 📖 Пример 1: Поиск информации

### Задача
Пользователь спрашивает: "Какая зарплата у бариста?"

### Решение

**Шаг 1:** Определяем тип вопроса
- Это вопрос о данных (не концепция, не процесс)
- → Ищем в `knowledge-base/`

**Шаг 2:** Определяем синонимы
- зарплата → з/п, оплата, зарплат

**Шаг 3:** Ищем файл
```bash
grep -r "з/п\|зарплат\|оплата" knowledge-base/ --include="*.csv" --include="*.md"
```

**Шаг 4:** Находим
- `knowledge-base/ДЛЯ ШЕФА/Таблица по З/п сотрудников.csv`

**Шаг 5:** Читаем и отвечаем
```
Зарплата бариста: 3500-5000 руб/смена (зависит от выручки)

Источник: knowledge-base/ДЛЯ ШЕФА/Таблица по З/п сотрудников.csv
```

## 🔨 Пример 2: Создание метода

### Задача
Создать метод приготовления латте.

### Решение

**Шаг 1:** Определяем стадию процесса
- Это извлечение метода
- → `process/07-method-and-product-extraction.md`

**Шаг 2:** Берём шаблон
```bash
cp PACK-cafe-operations/_template/03-methods/_method-card-template.md \
   PACK-cafe-operations/03-methods/CAFE.METHOD.003-latte.md
```

**Шаг 3:** Заполняем шаблон
```markdown
# CAFE.METHOD.003 - Приготовление латте

## ID
CAFE.METHOD.003

## Название
Приготовление латте

## Назначение
Приготовить латте согласно стандартам VK-offee

## Входы
- Эспрессо (1 или 2 шота)
- Молоко (200-250 мл)
- Чашка 300 мл

## Процесс
1. Приготовить эспрессо в чашку
2. Взбить молоко до температуры 60-65°C
3. Влить молоко в эспрессо
4. Создать латте-арт (опционально)

## Выходы
- [CAFE.WP.003 - Латте](../04-work-products/CAFE.WP.003-latte.md)

## Связанные различения
- [Стандарт ≠ Навык](../01-domain-contract/01B-distinctions.md#5-стандарт--навык)

## Failure modes
- [CAFE.FAIL.001 - Перегретое молоко](../05-failure-modes/CAFE.FAIL.001-burnt-milk.md)
```

**Шаг 4:** Создаём Work Product
```bash
cp PACK-cafe-operations/_template/04-work-products/_work-product-card-template.md \
   PACK-cafe-operations/04-work-products/CAFE.WP.003-latte.md
```

```markdown
# CAFE.WP.003 - Латте

## ID
CAFE.WP.003

## Название
Латте

## Описание
Кофейный напиток на основе эспрессо и молока

## Observability criteria
- Объём: 300 мл
- Температура: 60-65°C
- Соотношение эспрессо:молоко = 1:3
- Визуально: слои различимы

## Создаётся методом
- [CAFE.METHOD.003 - Приготовление латте](../03-methods/CAFE.METHOD.003-latte.md)

## Используется в
- Меню VK-offee
- Обучение бариста
```

**Шаг 5:** Обновляем индекс
```markdown
# PACK-cafe-operations/02-domain-entities/02C-methods-index.md

## Методы приготовления напитков

| ID | Название | Статус |
|----|----------|--------|
| CAFE.METHOD.001 | Эспрессо | Active |
| CAFE.METHOD.002 | Капучино | Active |
| CAFE.METHOD.003 | Латте | Active |
```

**Шаг 6:** Обновляем карту
```markdown
# PACK-cafe-operations/07-map/CAFE.MAP.001.md

## Связи методов

CAFE.METHOD.003 (Латте)
  → создаёт → CAFE.WP.003 (Латте)
  → использует → CAFE.METHOD.001 (Эспрессо)
  → может вызвать → CAFE.FAIL.001 (Перегретое молоко)
```

**Шаг 7:** Добавляем в SRT
```bash
# Латте — это процесс (F6)
ln -s ../../PACK-cafe-operations/03-methods/CAFE.METHOD.003-latte.md \
      SRT/F6-SoI-Manager/latte.md
```

**Шаг 8:** Запускаем process-lint
```bash
# Проверяем:
# ✅ Method имеет WP link
# ✅ Method имеет distinction link
# ✅ WP имеет observability criteria
# ✅ Карта обновлена
# ✅ Индекс обновлён
# ✅ Нет дидактического языка
```

**Шаг 9:** Коммитим
```bash
git add PACK-cafe-operations/03-methods/CAFE.METHOD.003-latte.md
git add PACK-cafe-operations/04-work-products/CAFE.WP.003-latte.md
git add PACK-cafe-operations/02-domain-entities/02C-methods-index.md
git add PACK-cafe-operations/07-map/CAFE.MAP.001.md
git add SRT/F6-SoI-Manager/latte.md
git commit -m "feat(pack): добавлен метод приготовления латте"
```

## 🔍 Пример 3: Анализ проблемы

### Задача
Бот на Railway не находит информацию о зарплате, хотя локально работает.

### Решение

**Шаг 1:** Проверяем bounded-context
- Зарплата входит в домен? → Да (ДЛЯ ШЕФА)

**Шаг 2:** Проверяем distinctions
- Какие различения применимы?
- → Калькуляция ≠ Себестоимость (но это не про зарплату)
- → Нет специального различения для зарплаты

**Шаг 3:** Проверяем данные
```bash
ls -la knowledge-base/ДЛЯ\ ШЕФА/
# Файл есть: Таблица по З/п сотрудников.csv
```

**Шаг 4:** Проверяем код бота
```python
# telegram-bot/bot.py
synonyms = {
    'зарплата': ['з/п', 'оплата', 'зарплат'],
}
```

**Шаг 5:** Проверяем логи Railway
```
Searching for: зарплата
Files checked: 299
Results found: 0
```

**Шаг 6:** Гипотеза
- Файл есть в git
- Синонимы настроены
- Но результатов 0
- → Возможно, проблема с кодировкой или путями

**Шаг 7:** Добавляем логирование
```python
print(f"Searching in: {file_path}")
print(f"File exists: {os.path.exists(file_path)}")
print(f"File content preview: {content[:100]}")
```

**Шаг 8:** Создаём Failure Mode
```markdown
# CAFE.FAIL.002 - Бот не находит файл на Railway

## Описание
Telegram-бот на Railway не находит файлы, которые есть в репозитории

## Detection test
1. Локально: бот находит файл
2. На Railway: бот не находит файл
3. Файл есть в git репозитории

## Причины
- Проблема с путями (относительные vs абсолютные)
- Проблема с кодировкой (UTF-8 vs CP1251)
- Файл не синхронизирован с Railway

## Связанные методы
- [Поиск в knowledge-base](../03-methods/CAFE.METHOD.XXX-search.md)

## Решение
1. Проверить пути в bot.py
2. Добавить логирование
3. Проверить кодировку файлов
```

## 📋 Пример 5: Применение различений

### Задача
Менеджер говорит: "Добавь в меню новый кофе"

### Решение

**Шаг 1:** Применяем различение "Меню ≠ Рецепт"
- Меню — что предлагаем клиенту (название, цена, описание)
- Рецепт — как готовим (ингредиенты, граммовки, технология)

**Шаг 2:** Уточняем у менеджера
- "Нужно добавить только в меню или также создать рецепт?"
- "Если рецепт, то какие ингредиенты и технология?"

**Шаг 3:** Если только меню
```markdown
# SRT/F2-Suprasystem-Engineer/menu-update.md

Добавить в меню:
- Название: "Раф кофе"
- Цена: 250 руб
- Описание: "Нежный кофе со сливками и ванилью"
```

**Шаг 4:** Если также рецепт
```markdown
# PACK-cafe-operations/03-methods/CAFE.METHOD.004-raf.md

## Процесс
1. Приготовить эспрессо
2. Добавить сливки 10%
3. Добавить ванильный сахар
4. Взбить капучинатором
```

**Шаг 5:** Связываем
- Меню (F2) ссылается на Рецепт (PACK)
- Рецепт (PACK) создаёт Work Product
- Work Product используется в Меню

## 🔗 Пример 7: Работа с FPF

### Задача
Определить, является ли "Смена" ролью или системой?

### Решение

**Шаг 1:** Читаем FPF
```
.fpf/INDEX.md → U.Role
.fpf/FPF-Spec.md → A.7 Strict Distinction
```

**Шаг 2:** Применяем принцип
- **Role** — маска, которую надевает Agent
- **System** — физический объект

**Шаг 3:** Анализируем "Смена"
- Смена — это период времени (8:00-16:00)
- Смена не физический объект
- Смена не маска для Agent

**Шаг 4:** Вывод
- Смена — это **Bounded Context** (ограниченный контекст времени)
- Внутри смены работают Agents в ролях (бариста, официант)

**Шаг 5:** Применяем различение
```markdown
# distinctions.md

## 2. Смена ≠ Сотрудник

- **Смена** — период работы с открытием и закрытием кассы
- **Сотрудник** — человек, который может работать в разных сменах

Процессы привязаны к смене, ответственность — к сотруднику.
```
