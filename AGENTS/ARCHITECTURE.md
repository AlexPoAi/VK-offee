# ARCHITECTURE - Архитектура FPF+SPF+SRT+FMT

Полное описание архитектуры VK-offee для AI-агентов.

---

## 🏛️ Уровень 1: FPF (First Principles Framework)

### Что это
**FPF** — "операционная система для мышления", трансдисциплинарная архитектура рассуждений от Анатолия Левенчука.

### Ключевые принципы

| Принцип | Паттерн | Применение в VK-offee |
|---------|---------|----------------------|
| **Холонический фундамент** | A.1 | Кофейня — холон: часть сети и целое само по себе |
| **Ограниченный контекст** | A.1.1 | Термины определены в границах домена кофеен |
| **Строгое различение** | A.7 | Роль ≠ Метод ≠ Работа. Меню ≠ Рецепт |
| **F-G-R доверие** | B.3 | Доверие = (Формальность, Область, Надёжность) |
| **Цикл ADI** | B.5 | Абдукция → Дедукция → Индукция |
| **Трансформеры** | A.7 | Agent → Method → Description → Work |

### Где находится
- `.fpf/FPF-Spec.md` (3.8 МБ, полная спецификация)
- `.fpf/FPF-Readme.md` (введение)
- `.fpf/INDEX.md` (индекс принципов)

### Когда использовать
- При создании новых процессов
- При определении ролей и границ
- При концептуальных проблемах
- При рефакторинге архитектуры

## 📋 Уровень 2: SPF (Second Principles Framework)

### Что это
**SPF** — методология создания Pack (формализованного доменного знания).

### 12 стадий процесса

| Стадия | Файл | Что делает |
|--------|------|-----------|
| **00** | process-overview.md | Обзор процесса |
| **01** | domain-selection.md | Выбор домена |
| **02** | bounded-context.md | Границы контекста |
| **03** | distinctions-work.md | Работа с различениями |
| **04** | domain-entities-identification.md | Идентификация сущностей |
| **05** | information-ingestion.md | Приём информации |
| **06** | analysis-and-formalization.md | Анализ и формализация |
| **07** | method-and-product-extraction.md | Извлечение методов |
| **08** | failure-modes-extraction.md | Извлечение ошибок |
| **09** | sota-annotation.md | SoTA-аннотации |
| **10** | map-maintenance.md | Поддержка карт |
| **11** | review-and-evolution-cycle.md | Цикл эволюции |

### Process Lint (обязательно!)
**Файл:** `process/process-lint.md`

**Проверяет:**
- Нет дидактического языка ("step", "lesson", "try this")
- Метод не сценарий
- Work product наблюдаем
- Failure modes типизированы
- Ссылки валидны
- Карты обновлены

### Hard Gates (блокируют коммит)

| Gate | Условие | Решение |
|------|---------|---------|
| HG-1 | Method без WP link | Добавить ссылку |
| HG-2 | Method без distinction | Добавить ссылку |
| HG-3 | WP без existence criteria | Добавить критерии |
| HG-4 | FM без detection test | Добавить тест |
| HG-5 | Structural change без map update | Обновить map |
| HG-6 | Didactic language | Переписать |

### Где находится
- `process/` (12 файлов + process-lint.md)
- `spec/` (спецификации для downstream)

## 🗂️ Уровень 3: SRT (Systems-Roles-Table)

### Что это
**SRT** — организация знаний через таблицу 3×3 (9 семейств документов F1-F9 + метасемейство F0).

### Таблица 3×3

```
                     Предприниматель  Инженер           Менеджер
                     (Зачем?)         (Как устроено?)   (Как работает?)
┌────────────────────────────────────────────────────────────────────┐
│ Suprasystem        F1               F2                F3           │
│ (Надсистема)       Контекст         Окружение         Взаимодействие│
│                    и рынок          и интерфейсы      и партнёрства│
├────────────────────────────────────────────────────────────────────┤
│ System-of-Interest F4               F5                F6           │
│ (Целевая система)  Требования       Архитектура       Реализация   │
│                    и ценность       продукта          и процессы   │
├────────────────────────────────────────────────────────────────────┤
│ Constructor        F7               F8                F9           │
│ (Система создания) Принципы         Платформа         Команда      │
│                    и экономика      и инструменты     и методология│
└────────────────────────────────────────────────────────────────────┘
```

### Три системы для VK-offee

| Система | Что это |
|---------|---------|
| **Suprasystem (F1-F3)** | Рынок HoReCa, клиенты, партнёры |
| **System-of-Interest (F4-F6)** | Сеть кофеен VK-offee |
| **Constructor (F7-F9)** | Команда менеджмента + инструменты |

### Семейства документов

| F | Система | Роль | Фокус | Примеры |
|---|---------|------|-------|---------|
| **F0** | Meta | - | Управление хранилищем | CLAUDE.md, отчёты |
| **F1** | Suprasystem | Entrepreneur | Контекст и рынок | Анализ рынка HoReCa |
| **F2** | Suprasystem | Engineer | Окружение | Меню, интерфейсы |
| **F3** | Suprasystem | Manager | Взаимодействие | Поставщики, франчайзинг |
| **F4** | SoI | Entrepreneur | Требования | Миссия, ценность |
| **F5** | SoI | Engineer | Архитектура | Планировка, оборудование |
| **F6** | SoI | Manager | Реализация | Процессы, рецепты |
| **F7** | Constructor | Entrepreneur | Принципы | Управление, экономика |
| **F8** | Constructor | Engineer | Платформа | Боты, GitHub, инструменты |
| **F9** | Constructor | Manager | Команда | Методология, протоколы |

### Где находится
- `SRT/` (10 папок: F0-F9, каждая с README.md)

### Как использовать
1. Определи, о какой системе идёт речь (Suprasystem/SoI/Constructor)
2. Определи роль (Entrepreneur/Engineer/Manager)
3. Найди соответствующее семейство (F0-F9)
4. Положи документ туда

## 🏗️ Уровень 4: PACK (Доменное знание)

### Что это
**PACK-cafe-operations** — формализованное доменное знание о кофейнях.

### Структура

```
PACK-cafe-operations/
├── 00-pack-manifest.md           # Манифест
├── 01-domain-contract/           # Контракт домена
│   ├── 01A-bounded-context.md    # Границы
│   ├── 01B-distinctions.md       # 12 различений
│   └── 01C-ontology.md           # Онтология
├── 02-domain-entities/           # Сущности
│   ├── 02A-roles.md              # Роли
│   ├── 02B-products.md           # Продукты
│   └── 02C-methods-index.md      # Индекс методов
├── 03-methods/                   # Методы
│   ├── CAFE.METHOD.001-espresso.md
│   └── ...
├── 04-work-products/             # Рабочие продукты
│   ├── CAFE.WP.001-menu.md
│   └── ...
├── 05-failure-modes/             # Типовые ошибки
│   ├── CAFE.FAIL.001-burnt-milk.md
│   └── ...
├── 06-sota/                      # SoTA-аннотации
│   └── CAFE.SOTA.001-coffee-extraction.md
├── 07-map/                       # Карты связей
│   └── CAFE.MAP.001.md
└── _template/                    # Шаблоны
```

### ID система

| Тип | Паттерн | Пример |
|-----|---------|--------|
| Method | `CAFE.METHOD.<NNN>-name.md` | `CAFE.METHOD.001-espresso.md` |
| Work Product | `CAFE.WP.<NNN>-name.md` | `CAFE.WP.001-menu.md` |
| Failure Mode | `CAFE.FAIL.<NNN>-name.md` | `CAFE.FAIL.001-burnt-milk.md` |
| SoTA | `CAFE.SOTA.<NNN>-name.md` | `CAFE.SOTA.001-extraction.md` |
| Map | `CAFE.MAP.<NNN>.md` | `CAFE.MAP.001.md` |

### 12 ключевых различений

1. **Меню ≠ Рецепт** (маркетинг vs операции)
2. **Смена ≠ Сотрудник** (период vs человек)
3. **Клиент ≠ Постоянный клиент** (разовый vs регулярный)
4. **Процесс ≠ Сервис** (внутреннее vs внешнее)
5. **Стандарт ≠ Навык** (обязательное vs дополнительное)
6. **Калькуляция ≠ Себестоимость** (теория vs факт)
7. **Должностная инструкция ≠ Чек-лист** (общее vs конкретное)
8-12. (см. `PACK-cafe-operations/01-domain-contract/01B-distinctions.md`)

## 🔧 Уровень 5: FMT (Практические протоколы)

### Что это
**FMT-exocortex-template** — практические протоколы работы с репозиториями.

### Протоколы

| Протокол | Назначение |
|----------|-----------|
| **Open-Work-Close** | Цикл работы с задачей |
| **Extraction** | Извлечение знаний из заметок |
| **Sync** | Синхронизация с внешними источниками |

### Связь с творческим конвейером

**Творческий конвейер** (`~/Documents/creativ-convector.nocloud/`) — система обработки заметок из Obsidian.

**Поток:**
```
Obsidian заметки
    ↓
Творческий конвейер (обработка)
    ↓
Извлечение знаний
    ↓
VK-offee PACK (формализация)
```

## 🔗 Связи между уровнями

```
FPF (принципы)
    ↓ определяет язык
SPF (процессы)
    ↓ создаёт
PACK (знания)
    ↓ организуется через
SRT (F0-F9)
    ↓ используется в
FMT (протоколы)
```

## 📊 Пример: Создание метода "Капучино"

1. **FPF:** Определяем, что это Method (не Tool, не Work)
2. **SPF:** Следуем стадии 07 (method-and-product-extraction)
3. **PACK:** Создаём `CAFE.METHOD.002-cappuccino.md` по шаблону
4. **SRT:** Размещаем в F6-SoI-Manager (процессы реализации)
5. **FMT:** Используем протокол Open-Work-Close
