### 1.1 Ингредиент
Партия сырья → `KITCHEN.ENTITY.004`

Физическое пространство производства → `KITCHEN.ENTITY.005`

## 2. Проверка полноты

✔ есть вход (ингредиент)
✔ есть промежуточное звено (заготовка)
✔ есть выход (блюдо)
✔ есть среда (цех)

👉 схема замкнута

---

## 3. Основная цепочка преобразования

```
Ингредиент → Заготовка → Блюдо → [передача] → Порция еды (PACK-cafe-operations)
```

---

## 4. Типы переходов

### 4.1 Преобразование (создание новой системы)
- ингредиент → заготовка
- заготовка → блюдо

✔ старая система исчезает (A.4)
✔ создаётся новая

### 4.2 Прямое использование
- ингредиент → блюдо (без заготовки)

### 4.3 Контекстный переход
- блюдо (PACK-kitchen) → порция еды (PACK-cafe-operations)

✔ физический объект тот же
✔ меняется BoundedContext (A.1.1), не нарушение A.7

---

## 5. Пространственная связь

Кухонный цех:
- содержит → ингредиенты
- содержит → заготовки
- является местом производства → блюда

---

## 6. Прослеживаемость партий (Traceability)

Каждая партия (ингредиент, заготовка, блюдо) имеет:
- уникальный идентификатор — часть системы (физика)
- физическую маркировку — часть системы (физика)
- связь с документом (накладная, сертификат) — U.Episteme (описание, не система)

Это делает модель управляемой:
- можно найти источник любой проблемы
- можно изолировать партии
- можно контролировать срок годности
- нельзя смешивать партии с разной маркировкой

---

## 7. Инварианты системы

- ингредиент не может стать блюдом без преобразования
- заготовка не существует после включения в блюдо
- блюдо не существует в PACK-kitchen после передачи на бар
- системы не смешиваются по стадиям

---

## 8. Критические различения

- объект ≠ процесс
- система ≠ состояние
- физика ≠ учёт (U.System ≠ U.Episteme)
- преобразование ≠ перемещение
- смена контекста ≠ уничтожение системы
