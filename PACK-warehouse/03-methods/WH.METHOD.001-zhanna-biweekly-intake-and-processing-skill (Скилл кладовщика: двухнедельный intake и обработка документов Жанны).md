# WH.METHOD.001 — Скилл кладовщика: двухнедельный intake и обработка документов Жанны

## Цель

Стабильно обрабатывать живые складские документы, которые Жанна выгружает **раз в 2 недели после инвентаризации**, и превращать их в управленческие решения без ручного хаоса.

## Режим работы агента

- **Только manual-mode** (по команде владельца).
- Триггер в чате: **«запусти кладовщика»**.
- После команды агент выполняет полный цикл intake + анализ + отчёт.

## Триггер запуска

1. Жанна подтверждает, что загрузила новый пакет документов.
2. Агент запускает обработку вручную (операционный режим):
```bash
cd ~/Github/VK-offee
bash PACK-warehouse/tools/warehouse_full_loop.sh
```

## Маршрут данных (as-is)

### Google Drive (source of intake)
- `Новое` (`WAREHOUSE_DRIVE_FOLDER_ID`) — Жанна кладёт свежие файлы.
- `Обработано` (`WAREHOUSE_DRIVE_PROCESSED_FOLDER_ID`) — скрипт переносит файлы после цикла.

### Локальный репозиторий (артефакты)
- `knowledge-base/Отчёты для бота/*` — исходники и копии для бота.
- `PACK-warehouse/02-domain-entities/report-cards/` — карточки отчётов.
- `PACK-warehouse/04-work-products/` — сводки, реестр, управленческие WP.

## Пошаговый протокол

### Шаг 1 — Pre-check перед запуском
1. Убедиться, что есть доступ к Google API (`token.pickle`, `credentials.json`).
2. Убедиться, что в `Новое` есть новые файлы (или подтвердить, что их нет).
3. Подтвердить текущую дату цикла (двухнедельное окно).

### Шаг 2 — Intake и техобработка
1. Запустить `warehouse_full_loop.sh`.
2. Проверить в логе:
   - сколько таблиц/файлов найдено;
   - есть ли ошибки;
   - были ли перемещения в `Обработано`.
3. Проверить `WH.REGISTRY.001-documents.csv`:
   - `new/processed/duplicate/error`.

#### Форматы входа
- Автоматически (as-is): `.xlsx`, `.csv`, Google Sheets.
- Частично вручную: `doc/docx/pdf` (если Жанна пришлёт в этих форматах, агент делает пометку о ручном извлечении данных и фиксирует gap на автоматизацию парсера).

### Шаг 3 — Аналитическая обработка
1. Для каждого нового источника сформировать карточку `WH.CARD.*`.
2. Сформировать управленческую аналитику по 5 блокам:
   - `Срочно заказать`: SKU, текущий остаток, сколько заказать, поставщик, дедлайн заказа.
   - `Цена закупа`: рост/снижение цен по накладным/каталогу за период.
   - `ABC + продажи`: топ продаж, слабые позиции, рекомендации по выводу/промо.
   - `Расход`: аномальный расход по моющим/хоз товарам.
   - `Риски данных`: чего не хватает для точного решения.
3. Если пришёл `ABC анализ*.xlsx/.csv`, добавить ABC-карточку и применить A/B/C в приоритизации закупки.

### Шаг 4 — Коммуникация и фиксация
1. Обновить `WH.REPORT.002` и/или выпустить новый `WH.WP.*`.
2. Отправить короткую Telegram-выжимку со ссылками на карточки.
3. Обновить `PACK-warehouse/DOCUMENT-REGISTRY.md`.

### Шаг 5 — Post-check
1. Нет ли необработанных файлов в `Новое`.
2. Нет ли новых `error` в реестре и DLQ.
3. Все новые документы привязаны к карточкам.

## Обязательный состав отчёта кладовщика

1. Что осталось критично (`<= min`) и что заказать сегодня (кол-во + поставщик + дедлайн).
2. Что реально продаётся: топ и слабые позиции (qty/выручка).
3. ABC-сегментация (A/B/C) по SKU и риск out-of-stock.
4. Динамика закупочных цен: что подорожало/подешевело.
5. Аномальный расход (моющие/хоз) и точки контроля.
6. Рекомендации по заказу/перераспределению/выводу из ассортимента.

## SLA цикла

1. **Время реакции:** не позже 2 часов после сигнала “Жанна загрузила”.
2. **Время до управленческой выжимки:** не позже 4 часов.
3. **Качество:** каждая новая поставка данных должна закончиться actionable карточкой.
