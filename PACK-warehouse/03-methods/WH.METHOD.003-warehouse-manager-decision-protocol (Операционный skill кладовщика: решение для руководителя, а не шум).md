# WH.METHOD.003 — Операционный skill кладовщика: решение для руководителя, а не шум

## Зачем

Этот метод фиксирует, что складской агент выдаёт не sync-сводку и не список файлов,
а управленческое решение.

## Главный принцип

Сверху всегда должен быть ответ на вопрос:
`что делать по складу сейчас`.

Не ответ на вопрос:
`какие документы были обработаны`.

## Верхний формат отчёта

1. `Executive summary`
2. `Срочно заказать`
3. `Планово заказать`
4. `Не заказывать / сократить`
5. `Проверить вручную`
6. `Каких данных не хватает`

## Обязательная логика

### `Срочно заказать`

Для каждой позиции:
- SKU
- текущий остаток
- сколько заказать
- поставщик
- контакт / канал
- дедлайн
- риск

### `Планово заказать`

Для каждой позиции:
- SKU
- горизонт заказа
- поставщик
- причина

### `Не заказывать / сократить`

Для каждой позиции:
- SKU
- почему стоп
- на какой период

### `Проверить вручную`

Только сюда выносятся:
- missing contacts
- спорные supplier matches
- неполный ABC
- аномальная маржа
- плохое качество данных

## Запрещено в верхнем слое

- список `WH.CARD.*`
- список `csv/pdf/md`
- технические статусы sync/pipeline как главный смысл отчёта
- длинный перечень ссылок без действия

## Временная operational формула количества заказа

До уточнения полной модели `par level / reorder point` использовать консервативное правило:
- минимальный целевой уровень: `10 единиц на категорию`
- для кофе, дрипов, чая и шоколада не опускаться ниже `10`
- если позиция ниже минимума, дозаказывать до `10`, если нет более сильного доменного правила

Формула:
`quantity_to_order = max(0, 10 - current_stock)`

С обязательной пометкой, что это временная operational heuristic, подтверждённая владельцем 20.04.2026.

## Доменные уточнения владельца

- дрип-кофе: поставщик `Тэйсти Кофе`, также есть `Субмарина`
- чай: поставщик `Тэйсти Кофе`
- шоколад: `UNICAVA`
- `lead time` для `Тэйсти Кофе`, `Субмарина`, `UNICAVA`: `10-12 дней`
- если по поставщику не хватает накладных, кладовщик обязан явно вывести это в блок `Проверить вручную`
- десерты и кухня (`Дмитрий`, блины, роллы, киши и т.п.) не выводятся как direct reorder-задача владельцу
- по десертам и еде агент ведёт только: `рост/снижение цен от поставки к поставке` и `анализ продаж с рекомендацией оставить / сокращать / снимать`

## Источники решения

- остатки -> дефицит / избыток
- продажи -> движение / слабые позиции
- ABC -> приоритет
- накладные -> поставщик / закупочный контур / изменение цен
- supplier cards -> контакт / канал заказа / контекст поставщика

## Quality Gate

- [ ] Нет шумовых артефактов в executive layer
- [ ] У критичных позиций есть `supplier + qty + deadline`
- [ ] Есть блок `Не заказывать / сократить`
- [ ] Есть блок `Каких данных не хватает`
- [ ] Telegram работает как доставка, а не как место формирования смысла
