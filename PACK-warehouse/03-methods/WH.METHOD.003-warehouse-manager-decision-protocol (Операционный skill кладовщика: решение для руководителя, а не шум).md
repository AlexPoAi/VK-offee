# WH.METHOD.003 — Операционный skill кладовщика: решение для руководителя, а не шум

## Зачем

Этот метод фиксирует, что складской агент выдаёт не sync-сводку и не список файлов,
а управленческое решение.

## Главный принцип

Сверху всегда должен быть ответ на вопрос:
`что делать по складу сейчас`.

Не ответ на вопрос:
`какие документы были обработаны`.

Руководительский слой обязан быть:
- коротким;
- на русском;
- без инженерного шума и внутренних кодов артефактов.

Запрещено поднимать в верхний слой:
- `WH.REPORT.*`, `WH.SESSION.*`, `WH.CARD.*`;
- английские заголовки;
- matched/unmatched, sheet-name, paths, counters парсинга;
- технические слова без короткого пояснения.

## Верхний формат отчёта

1. `Executive summary`
2. `Срочно заказать`
3. `Планово заказать`
4. `Не заказывать / сократить`
5. `Проверить вручную`
6. `Каких данных не хватает`

## Обязательная логика

### `Язык руководителя`

Если термин важен, но неочевиден, агент обязан:
- либо заменить его на простой русский эквивалент;
- либо коротко объяснить его в одной строке.

Пример:
- `позиции со слабой маржой`
- пояснение:
  `это товары, где себестоимость съедает слишком большую часть выручки`

### `Нормализация входа`

До формирования решения агент обязан:
- выбросить битые SKU и mojibake из executive layer;
- нормализовать человеческие названия товаров;
- не поднимать в отчёт строки вида `????`, `капельница`, обрезанные коды и иные артефакты OCR/CSV.

Если строка не нормализуется уверенно, она попадает только в `Проверить вручную`.

### `Срочно заказать`

Для каждой позиции:
- SKU
- текущий остаток
- сколько заказать
- поставщик
- контакт / канал
- дедлайн
- риск

### `Планово заказать`

Для каждой позиции:
- SKU
- горизонт заказа
- поставщик
- причина

### `Не заказывать / сократить`

Для каждой позиции:
- SKU
- почему стоп
- на какой период

### `Проверить вручную`

Только сюда выносятся:
- missing contacts
- спорные supplier matches
- неполный ABC
- аномальная маржа
- плохое качество данных
- битые названия SKU
- неоднозначный supplier routing по drip-позициям

## Запрещено в верхнем слое

- список `WH.CARD.*`
- список `csv/pdf/md`
- технические статусы sync/pipeline как главный смысл отчёта
- длинный перечень ссылок без действия
- битые SKU и нечитабельные позиции

## Operational формула количества заказа

До уточнения полной модели `par level / reorder point` использовать подтверждённые владельцем целевые остатки:
- чай `100 г` для `Ков Маркет`: целевой остаток `15 пачек`
- чай `250 г / НА БАР`: целевой остаток `6 пачек`
- зерно `250 г`: целевой остаток `30 пачек`
- зерно `1 кг`, кроме бункерного зерна: целевой остаток `15 кг`
- для остальных категорий без доменного правила временно использовать fallback `10 единиц`

Отдельный будущий слой: для самых покупаемых SKU зерна `250 г` по `ABC / продажи` кладовщик должен предлагать повышенный target, например `50 пачек`, но только после явной проработки и подтверждения владельцем. До подтверждения все `250 г` считать по базовой цели `30 пачек`.

Формула:
`quantity_to_order = max(0, target_stock - current_stock)`

Если SKU попадает в исключение (`бункер`, спорная фасовка, неясное название), не считать заказ автоматически: вынести в блок `Проверить вручную` или предложить перемещение между точками.

## Целевые показатели заказа

Каждый оформленный заказ кладовщик обязан фиксировать как отдельный order artifact, а не только как файл.

Обязательные показатели по заказу:
- поставщик / группа поставщиков
- дата оформления
- статус: `черновик`, `готово к отправке`, `оформлено`, `получено`, `частично получено`
- ссылка на локальную копию файла
- ссылка на Google Sheets, если файл загружен в Drive
- строк заказа всего
- сумма заказа всего
- скидка от базового прайса, если применима
- обычная рабочая скидка для сравнения, если известна
- условия анализа цены: сравнивать цену только после нормализации по скидке

Обязательные показатели по SKU в заказе:
- `current_stock`: текущий остаток перед заказом
- `target_stock`: целевой остаток по правилу категории
- `recommended_qty`: расчётная рекомендация кладовщика
- `ordered_qty`: фактически заказанное количество
- `order_delta`: `ordered_qty - recommended_qty`
- `unit_price`: закупочная цена в заказе
- `line_total`: сумма строки
- `abc_class`, если есть
- причина отклонения от рекомендации: `ABC`, ручное решение владельца, промо, скидка, новый SKU, замена, нет в ассортименте

Целевые правила для контроля заказа:
- если `ordered_qty < recommended_qty`, кладовщик помечает риск недозаказа
- если `ordered_qty > recommended_qty`, кладовщик помечает причину: например `ABC A`, повышенная цель, скидка, ручное решение
- если скидка отличается от обычной, кладовщик не делает вывод о росте/падении закупочной цены без пересчёта к единому discount basis
- для заказа `Тэйсти Кофе` от 2026-04-26 discount basis: факт `30%`, обычное условие `35%`

## Порядок выдачи заявки по прайсу

Кладовщик обязан выдавать позиции в том порядке, в котором владелец собирает заказ по прайсу, а не по алфавиту.

Текущий порядок для зерна:
1. `Декаф` / `ДКФ` / `Decaf`: декафеинизированный кофе.
2. Линейка `Milk`: `Брауни`, `Бэрри`, `Кэнди`, `Натти`, `Хани`.

Голосовой ввод нормализовать в русские SKU:
- `Candy` -> `Кэнди`
- `Natty` -> `Натти`
- `Honey` / `Hani` -> `Хани`
- `Berry` / `Barry` -> `Бэрри`
- `Brownie` -> `Брауни`

## Доменные уточнения владельца

- дрип-кофе: поставщик `Тэйсти Кофе`, также есть `Субмарина`
- чай: поставщик `Тэйсти Кофе`
- шоколад: `UNICAVA`
- для анализа цен по кофе кладовщик обязан учитывать скидку заказа от базового прайса; заказ от 2026-04-26 по `Тэйсти Кофе` был со скидкой `30%`, обычное рабочее условие — `35%`
- `lead time` для `Тэйсти Кофе`, `Субмарина`, `UNICAVA`: `10-12 дней`
- если по поставщику не хватает накладных, кладовщик обязан явно вывести это в блок `Проверить вручную`
- десерты и кухня (`Дмитрий`, блины, роллы, киши и т.п.) не выводятся как direct reorder-задача владельцу
- по десертам и еде агент ведёт только: `рост/снижение цен от поставки к поставке` и `анализ продаж с рекомендацией оставить / сокращать / снимать`
- `Тэйсти Кофе` и `Субмарина` нельзя схлопывать в одного поставщика по умолчанию; при сомнении строка идёт в ручную проверку
- `WH.SESSION.001` должен быть доской решений для руководителя, а не списком документов

## Источники решения

- остатки -> дефицит / избыток
- продажи -> движение / слабые позиции
- ABC -> приоритет
- накладные -> поставщик / закупочный контур / изменение цен
- supplier cards -> контакт / канал заказа / контекст поставщика

## Document-to-Decision Contract

Для каждого входящего типа документа кладовщик обязан держать цепочку:
`document -> extracted fields -> derived metrics -> manager output`.

Запрещено считать документ “обработанным”, если есть только:
- карточка файла;
- список колонок;
- список строк;
- техническая выжимка без управленческого вывода.

Документ считается доведённым только когда:
1. понятно, какие поля из него извлечены;
2. понятно, какие метрики из них посчитаны;
3. понятно, какой именно блок в manager-report из этого появился.

### `ABC-анализ`

- извлекаем:
  - SKU
  - категория `A/B/C`
  - продажи в штуках
  - выручка
  - доля
  - себестоимость, если присутствует
- считаем:
  - лидеры спроса
  - слабые SKU
  - риск заморозки денег
  - что расширять / не расширять / сокращать
- в отчёт руководителю:
  - `лидеры`
  - `слабые позиции`
  - `что не докупать`
  - `где деньги застряли`

### `Остатки`

- извлекаем:
  - SKU
  - текущий остаток
  - точка
  - группа товара
- считаем:
  - `critical stock`
  - `planned reorder`
  - `reorder qty`
  - `stop-sale risk`
- в отчёт руководителю:
  - `что заказать срочно`
  - `что заказать планово`
  - `где дефицит`

### `Накладные PDF`

- извлекаем:
  - поставщик
  - дата
  - SKU
  - закупочная цена
  - количество
  - сумма
- считаем:
  - `price delta`
  - `last purchase price`
  - историю закупки по поставщику
  - базу для карточек себестоимости
- в отчёт руководителю:
  - `что подорожало`
  - `что подешевело`
  - `у какого поставщика меняются условия`

### `Каталог`

- извлекаем:
  - SKU
  - текущая цена продажи
  - себестоимость / маржа, если присутствуют
- считаем:
  - `price mismatch`
  - отклонение цены
  - потенциал маржи
- в отчёт руководителю:
  - `где плохая цена`
  - `что нужно пересчитать`

### `Продажи / выручка`

- извлекаем:
  - SKU
  - штуки
  - выручка
  - период
  - точка
- считаем:
  - спрос
  - weak movers
  - top movers
  - различия по точкам
- в отчёт руководителю:
  - `что реально продаётся`
  - `что висит мёртвым грузом`

## Truthful usage rule

Если кладовщик не может для любого документа коротко ответить на 3 вопроса:
- `что я извлёк?`
- `что я посчитал?`
- `что увидел руководитель?`

значит document-chain ещё не доведён до идеала и считается только частично завершённым.

## Cost layer

Следующий обязательный слой развития кладовщика:
- карточки себестоимости по SKU / товарным группам;
- связка `закупочная цена -> цена продажи -> базовая маржа`;
- manager-ready вывод, что продаётся, но почти не зарабатывает, а что выгодно расширять.

## Quality Gate

- [ ] Нет шумовых артефактов в executive layer
- [ ] У критичных позиций есть `supplier + qty + deadline`
- [ ] Есть блок `Не заказывать / сократить`
- [ ] Есть блок `Каких данных не хватает`
- [ ] Telegram работает как доставка, а не как место формирования смысла
- [ ] В executive layer нет битых SKU и document-noise
- [ ] Десерты и кухня остались в аналитике, а не в direct reorder
