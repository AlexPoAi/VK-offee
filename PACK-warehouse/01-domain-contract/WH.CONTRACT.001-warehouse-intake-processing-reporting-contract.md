# WH.CONTRACT.001 — Контракт цикла склада: intake, processing, reporting

## Назначение
Определяет обязательные этапы, входы/выходы, роли и требования качества для складского цикла.

## 1) Intake from Google Drive `Новое`

### Вход
- Папка Google Drive `Новое` (операционный вход от Жанны).
- Поддерживаемые типы: `xlsx`, `xls`, `csv`, `pdf`.

### Операции
1. Синхронизация файлов из `Новое` в локальный контур.
2. Присвоение `doc_id` и запись в реестр.
3. Валидация минимальных метаданных (`filename`, `period/date`, `source`).

### Выход
- Зарегистрированные источники в `WH.REGISTRY.001-documents.csv`.
- Сырые файлы доступны для классификации.

## 2) Классификация типов документов

### Обязательные классы
- `stock` — остатки/инвентаризация.
- `sales` — продажи.
- `revenue` — выручка.
- `invoice_pdf` — накладные в PDF.
- `abc` — ABC-анализ.

### Дополнительные оси классификации (обязательно)
- `product_type` — тип продукции: `кофе_зерно`, `кофе_drip`, `чай`, `шоколад`, `расходники`, `другое`.
- `supplier` — поставщик из накладной/справочника.

### Правила
- Один документ должен иметь ровно один primary class.
- Неопределённый тип -> `unknown` и маршрут в DLQ с причиной.

## 3) Методы обработки по типам

### `stock` (остатки)
- Нормализация SKU/точек.
- Поиск дефицита (низкий остаток) и избытка.
- Выход: карточки рисков по SKU и точкам.
- SOTA-усиление:
  - `par_level` по SKU/точке;
  - `reorder_point` = спрос на lead time + safety stock;
  - `days_on_hand` / покрытие запасом;
  - `FEFO` для скоропорта (first-expired-first-out).

### `sales` (продажи)
- Агрегация объёма продаж по SKU/категориям.
- Сигналы роста/падения спроса по периоду.
- Выход: карточка динамики продаж.
- SOTA-усиление:
  - `ABC/XYZ` сегментация: важность по обороту + стабильность спроса;
  - `velocity` по SKU и по точкам;
  - `dead stock` / slow movers.

### `revenue` (выручка)
- Агрегация выручки по SKU/точкам/категориям.
- Выделение наиболее значимых вкладов.
- Выход: карточка денежной структуры.
- SOTA-усиление:
  - вклад в gross profit по группе;
  - price/margin drift по периодам;
  - концентрация выручки на top-SKU.

### `invoice_pdf` (накладные PDF)
- Извлечение строк: SKU, количество, цена, поставщик, дата.
- Проверка на полноту и читабельность.
- Выход: структурированная карточка поставок/закупок.
- Обязательная нормализация:
  - `supplier_name`
  - `product_type`
  - `sku_or_item`
  - `unit`
  - `qty`
  - `purchase_price`
  - `invoice_date`
- SOTA-усиление:
  - `supplier_price_history`;
  - `lead_time_history`;
  - `fill_rate` (что заказали vs что реально приехало);
  - сигнал роста цены закупа.

### `abc` (ABC)
- Чтение классов A/B/C.
- Кросс-сверка с остатками и продажами.
- Выход: матрица приоритетов закупки/контроля.

### `waste` / `variance` (обязательный future layer)
- `wastage`: списания, бой, просрочка, staff error.
- `actual_vs_theoretical`: сравнение фактического расхода и теоретического по продажам.
- Выход: карточка потерь, недостач и зон утечки денег.

## 4) Где хранится результат
- Карточки артефактов: `02-domain-entities/report-cards/`.
- Реестр документов: `04-work-products/WH.REGISTRY.001-documents.csv`.
- Сводка цикла: `04-work-products/WH.REPORT.002-warehouse-sync-summary-latest.md`.
- Очередь решений: `04-work-products/WH.SESSION.001-decision-queue-latest.md`.
- Некачественные/ошибочные файлы: quarantine/DLQ.

## 5) Из чего состоит отчёт руководителя

### Обязательные разделы
1. `Executive summary` (verdict + 3 ключевых вывода).
2. `Срочно (до 24ч)` (конкретные действия).
3. `Планово (2-7 дней)` (приоритетные задачи).
4. `Что не заказывать/сократить`.
5. `Риски данных` (чего не хватает и что запросить).
6. `План закупки по поставщикам` (обязательно).

### Раздел `План закупки по поставщикам` (обязательный)
Для каждой группы товаров:
- `тип товара`
- `что заканчивается`
- `сколько заказать`
- `поставщик`
- `контакт поставщика`
- `дедлайн заказа`
- `целевая дата поставки`
- `основание решения` (`остаток`, `продажи`, `ABC/XYZ`, `lead time`, `сезонность`)
- `что будет если не заказать`

### Обязательные decision-поля
- `decision_id`
- `sku_or_group`
- `product_type`
- `supplier_name`
- `supplier_contact`
- `action`
- `owner`
- `order_deadline`
- `delivery_eta`
- `reorder_point_basis`
- `lead_time_days`
- `days_on_hand`
- `due_date`
- `priority`
- `expected_effect`
- `status`
- `evidence_link`

## 6) Какие параметры/данные обязательны для качества

### Обязательные поля в данных
- Единый `SKU_ID` во всех файлах.
- `Дата/период` отчёта.
- `Точка/склад`.
- `Количество` и `единица измерения`.
- Для закупок: `цена`, `поставщик`, `дата поставки`.
- Для руководительского закупочного отчёта: `контакт поставщика`, `тип товара`, `минимальный остаток`, `целевой остаток`.
- Для SOTA-слоя:
  - `lead_time_days`
  - `waste_reason`
  - `actual_usage`
  - `theoretical_usage`
  - `fill_rate`
  - `expiry_date` / `days_to_expiry` для скоропорта

### Минимальный SLA качества
- Ошибки классификации <= 5% документов за цикл.
- Доля пустых/нечитаемых карточек = 0%.
- Каждое критичное действие в отчёте имеет `owner` и `due_date`.

## Кто делает
- Агент/роль: `Warehouse Demand Analyst (Кладовщик-аналитик)`.
- Ответственный за загрузку данных: `Жанна`.
- Ответственный за принятие решений: руководитель/владелец.

## Критерии приемки
1. Полный путь `Новое -> обработка -> отчёт -> Обработано` выполняется без потери файлов.
2. По каждому файлу есть либо карточка, либо запись в DLQ с причиной.
3. Руководительский отчёт содержит обязательные разделы и decision-поля.
4. Выгрузка в Telegram содержит краткий digest и рабочие ссылки на source-of-truth.
5. Для топ-рисков в отчёте есть объяснение не только `что мало`, но и `почему именно сейчас надо заказать`.

## Границы ответственности
- Агент отвечает за обработку, аналитику, структурирование и прозрачность.
- Агент не принимает финальные бизнес-решения и не подменяет владельца процесса.
- Агент фиксирует дефицит данных и инициирует запросы на дообогащение, но не создаёт первичку вместо источника.
