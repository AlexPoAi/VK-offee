# WH.WP.005 — Структура управленческого отчёта и протокол работы кладовщика

## 1) Цель
Сначала зафиксировать, **какой отчёт нужен руководителю**, и только потом настраивать доставку в Telegram.

Отчёт должен отвечать на 4 вопроса:
1. Что происходит со складом и продажами за период?
2. Где риск денег/дефицита/перезатарки?
3. Что делать сегодня и кто ответственный?
4. Каких данных не хватает для более точных решений?

## 2) Что считаем уже сейчас (из текущих данных)

### Данные, которые уже есть
- Остатки по складам (инвентаризация)
- Продажи (период)
- Выручка (период)
- Каталог (SKU + цена)
- ABC анализ (когда загружен)
- Накладные (часто PDF, частично структурно)

### Метрики уровня руководителя (можно считать сейчас)
1. Количество SKU в риске дефицита (`<= threshold`).
2. Top-дефицитные позиции (по точкам).
3. Top-позиции продаж (по объёму, где данные позволяют).
4. Сигналы перезатарки (избыток на складе).
5. A/B/C риск-матрица (если есть ABC).
6. Action-list: срочно заказать / планово заказать / сократить заказ.

## 3) Чего не хватает для «сильной» аналитики
1. Списания/брак по SKU и причине.
2. Факт поставок (`заказано`, `поставлено`, `дата поставки`, `lead time`).
3. Накладные в табличном SKU-формате (не только PDF).
4. Единый `SKU_ID` во всех файлах.
5. Минимальный/целевой остаток по SKU (policy-level данные).

## 4) Целевая структура отчёта (канон)

### Блок A — Executive summary (для владельца)
- Вердикт: `зелёный / жёлтый / красный`.
- 3 главных риска периода.
- 3 главных действия на сегодня.
- Потенциальный эффект (деньги/дефицит/оборачиваемость).

### Блок B — Решения на сегодня (action table)
Колонки:
- `SKU/категория`
- `Проблема`
- `Решение`
- `Срок`
- `Ответственный`
- `Статус (open/done)`

### Блок C — Диагностика склада и продаж
- Критичные остатки.
- Избыточные остатки.
- Движение спроса (на базе продаж/выручки).
- Разбивка по точкам.

### Блок D — Данные и качество
- Что было загружено в этот цикл.
- Какие файлы не обработались (DLQ/errors).
- Какие данные отсутствуют и кого запросить.

### Блок E — Follow-up к следующему циклу
- Что проверить через 3/7/14 дней.
- Какие гипотезы на улучшение ассортимента.

## 5) Формат вывода

### Полный отчёт (source of truth)
- `PACK-warehouse/04-work-products/WH.WP.*.md`
- Полная аналитика + таблица решений.

### Telegram (управленческий digest)
- Короткая версия: verdict + 3 действия + ссылка на полный отчёт.
- Без “технического шума”.

## 6) Operating protocol кладовщика (report-first)
1. Intake файлов от Жанны.
2. Техобработка и карточки.
3. Сбор метрик по канону (A→E).
4. Формирование action-table с owner+date.
5. Публикация full report.
6. Отправка короткого digest в Telegram.

## 7) KPI качества отчёта
1. В каждом отчёте есть минимум 3 конкретных действия.
2. У каждого действия есть owner и срок.
3. Нет “пустых” карточек без управленческого смысла.
4. В каждом цикле есть блок `data gaps` и запрос недостающих данных.
