# WH.WP.006 — Архитектура полного цикла кладовщика

## Цель
Зафиксировать единый сквозной цикл: от поступления файлов в Google Drive до управленческого отчёта руководителю с конкретными решениями.

## Контекст и границы
- Вход: документы Жанны в Google Drive папке `Новое`.
- Выход: структурированный управленческий отчёт + очередь решений + карточки артефактов.
- Telegram используется только как канал доставки digest и ссылок на source-of-truth.

## Полный цикл (E2E)

### Этап 1. Intake и регистрация входящих
- Проверка папки `Новое` в Google Drive.
- Скачивание/синхронизация в локальный контур.
- Присвоение `doc_id`, фиксация в реестре (`WH.REGISTRY.001-documents.csv`).
- Идемпотентность: повторная загрузка не должна ломать реестр и отчёты.

### Этап 2. Классификация документов
- Определение типа: `остатки`, `продажи`, `выручка`, `накладные PDF`, `ABC`.
- Проверка обязательных атрибутов: дата, точка, период, формат таблицы.
- Невалидные/нечитаемые документы -> quarantine/DLQ.

### Этап 3. Обработка по типам
- `Остатки`: расчёт дефицита/перезатарки по SKU и точкам.
- `Продажи`: объём и динамика по SKU/категориям за период.
- `Выручка`: денежный вклад и концентрация выручки.
- `Накладные PDF`: извлечение позиций, цен и поставщиков в структурированный вид.
- `ABC`: приоритизация SKU по A/B/C с кросс-проверкой остатков.
- Доп. слой: нормализация `тип товара -> поставщик -> контакт поставщика`.
- SOTA-слой:
  - `ABC/XYZ` по SKU;
  - `par level` и `reorder point`;
  - `lead time` и `supplier reliability`;
  - `waste/variance`;
  - `FEFO` для скоропорта;
  - `dead stock` / slow movers.

### Этап 4. Нормализация и хранение результатов
- Карточки по источникам -> `02-domain-entities/report-cards/`.
- Сводка цикла -> `04-work-products/WH.REPORT.002-warehouse-sync-summary-latest.md`.
- Очередь решений -> `04-work-products/WH.SESSION.001-decision-queue-latest.md`.
- Все исходники после обработки перемещаются `Новое -> Обработано`.

### Этап 5. Формирование отчёта руководителя (manager report)
- Формируется структурированный отчёт по секциям:
1. Вердикт периода (`green/yellow/red`) и 3 ключевых риска.
2. Срочные решения на 24 часа.
3. Плановые решения на 2-7 дней.
4. Что не заказывать/сократить.
5. Риски данных (чего не хватает для точных решений).
6. План закупки по поставщикам (тип товара, что заказать, кому, до какого числа).
7. Потери и утечки денег (`waste`, `variance`, price drift) — когда данные доступны.
- К каждому решению добавляются decision-поля (см. ниже).

### Этап 6. Доставка и follow-up
- Telegram digest: краткая выжимка + ссылки на полный отчёт и очередь решений.
- Дальше работа ведётся по очереди решений: назначение ответственных, сроки, статусы.

## Reporting Contract (обязательный состав управленческого отчёта)
- `report_id`, `report_period`, `generated_at`.
- `verdict` (green/yellow/red) + обоснование.
- `urgent_actions[]` (до 24ч).
- `planned_actions[]` (2-7 дней).
- `stop_or_reduce[]` (что не заказывать/сократить).
- `data_gaps[]` (недостающие данные/риски качества).
- `supplier_procurement_plan[]` (закупка по поставщикам и типам товаров).
- `loss_and_variance_signals[]` (потери, расхождения, рост цены закупа).
- `source_links[]` (рабочие ссылки на source-of-truth).

## Обязательные decision-поля (на каждую управленческую рекомендацию)
- `decision_id`
- `sku_or_group`
- `product_type`
- `supplier_name`
- `supplier_contact`
- `problem`
- `action`
- `priority` (`critical/high/normal`)
- `owner`
- `order_deadline`
- `delivery_eta`
- `why_now`
- `risk_if_no_action`
- `due_date`
- `expected_effect` (деньги/дефицит/оборачиваемость)
- `status` (`open/in_progress/done`)
- `evidence_link` (карточка/отчёт)

## Кто делает
- Агент: `Warehouse Demand Analyst (Кладовщик-аналитик)`.
- Роль агента: intake, классификация, аналитика, подготовка manager report, формирование очереди решений.
- Ответственный за данные: `Жанна` (поставка исходных документов по регламенту).
- Ответственный за управленческие решения: владелец/операционный руководитель.

## Лучшие практики, которые берём в target-model
1. Заказывать не по ощущению, а по `reorder point` и `lead time`.
2. Держать `par levels` по точкам, а не один общий остаток.
3. Разделять `A/X` позиции, где нельзя допускать stockout, и `C/Z`, где можно снижать запас.
4. Считать `actual vs theoretical usage`, чтобы видеть потери, бой и ошибки учёта.
5. Вести supplier scorecard: цена, стабильность, скорость, недопоставки.
6. Для скоропорта использовать `FEFO`, а не просто FIFO.

## Критерии приемки
1. После запуска цикла все валидные файлы из `Новое` отражены в реестре и карточках.
2. Невалидные файлы отражены в DLQ с причиной.
3. Управленческий отчёт содержит все обязательные секции и decision-поля.
4. В отчёте есть минимум 3 конкретных действия с `owner` и `due_date`.
5. В отчёте есть хотя бы 1 заполненный блок `поставщик -> товары -> дедлайн заказа`.
6. Telegram digest содержит только короткую выжимку и рабочие ссылки.
7. Исходники перемещены в `Обработано` без потери трассировки.

## Границы ответственности
- В зоне агента:
  - обработка складских документов и аналитика по наличию/движению;
  - формирование отчёта и очереди решений;
  - фиксация gaps и запросов на дообогащение данных.
- Вне зоны агента:
  - финальное управленческое решение по закупке/ассортименту;
  - изменение оргпроцессов без согласования владельца;
  - исправление первичных ошибок учёта в исходной системе (только сигнализация и запрос корректировок).
