> Разработана 26.03.2026 в ходе FPF-инспекции с FPF-консультантом.
> Следующий шаг: детализация PACK-bar и PACK-service → написание domain-contract для каждого PACK.

## Архитектура базы знаний

```
VK-offee (база знаний)
│
├── PACK-cafe-operations   ← контейнер домена «кофейня»
├── PACK-bar               ← поддомен (бар)
├── PACK-kitchen           ← отдельный PACK (кухня)
├── PACK-service           ← поддомен (сервис)
├── PACK-management        ← поддомен (управление)
├── PACK-hr                ← поддомен (персонал)
└── PACK-park-development  ← отдельный PACK (другой домен)
```

**Ключевое различение:**
- Это **декомпозиция знаний**, не иерархия систем
- PACK-kitchen связан с cafe-operations через объекты (поставка), не через подчинение

## PACK-cafe-operations

### Тип: кофейная точка

**Атрибуты экземпляра:**
- адрес
- режим работы
- состав оборудования
- вместимость

**Экземпляры:**
- Самокиша
- Тургенева
- Луговая

**Части (физические зоны):**
- бар → PACK-bar
- зал → PACK-service
- витрина (состояние остатков)
- холодильник (резерв/буфер)

### Объекты PACK-cafe-operations

#### [тип] заказ
```
заказ
  ├── клиент               → [клиент]
  ├── сотрудники [1..n]    → ссылка из PACK-hr
  ├── напитки [1..n]       → ссылка из PACK-bar
  ├── еда [0..n]           → с витрины (ссылка из PACK-kitchen)
  ├── оплаты [1..n]
  │   └── оплата: сумма + способ (нал | карта)
  ├── состояние: новый → в работе → готов → выдан
  └── время (характеристики, A.17 — не объект)
      ├── создание
      ├── начало
      ├── готовность
      └── выдача
```

Производные метрики:
- время приготовления = готовность − начало
- время ожидания = выдача − создание

Время фиксирует: система Saby (часть метода, не объект).

#### [тип] клиент
```
клиент
  └── заказы [0..n]
```
Сейчас: анонимный допустим. Атрибуты (имя, телефон) — добавить позже.

#### [тип] оплата
```
оплата
  ├── сумма
  └── способ: нал | карта
```
Один заказ → оплаты [1..n] (разбитая оплата, предоплата + доплата).

#### [тип] поставка
```
поставка
  ├── источник → [кухонный цех]  (создаёт кухня)
  ├── содержит → порции еды [1..n]
  └── приходит в → [кофейная точка]
```
Поставка → пополняет холодильник → ротация → витрина.

### Методы PACK-cafe-operations
- **ротация** — перемещение порций из холодильника на витрину (метод, не объект)

## PACK-kitchen

### Тип: кухонный цех
```
кухонный цех
  ├── атрибут: внутренний | внешний
  ├── прайс
  ├── номенклатура
  └── части: зона приготовления / упаковки / хранения
```

Экземпляры: наша кухня (внутренний) + поставщики (внешние).
Каждый поставщик = отдельный объект со своим прайсом и номенклатурой.

### Объекты PACK-kitchen

#### [тип] порция еды
```
порция еды
  ├── название
  └── описание
  (НЕТ ссылки на кухню — связь через поставку)
```
Создаётся кухней → передаётся через поставку → хранится в точке (витрина/холодильник).

#### [тип] заготовка
Полуфабрикат. Живёт внутри кухни, не выходит наружу.

## Правила размещения объектов (выведены в сессии)

**Правило 1:** Если объект проходит через несколько поддоменов → живёт в родительском PACK
> `заказ` → PACK-cafe-operations

**Правило 2:** Объект живёт там, где создаётся как тип
> `порция еды` → PACK-kitchen; `сотрудник` → PACK-hr

**Правило 3:** Атрибут на типе — не новый PACK
> Внешний поставщик = экземпляр `кухонного цеха` с атрибутом `внешний`

**Правило 4:** Не загрязнять объект связями — выносить в объект-посредник
> `порция еды` не хранит кухню → связь через `поставку`

**Правило 5:** Действие → метод (03-methods), состояние → объект/часть
> `ротация` = метод, не объект

**A.7:** Не дублировать описание типа в разных PACK — только ссылка
**A.17:** Время — характеристика объекта, не отдельный объект

## Полный поток еды

```
кухонный цех → создаёт порции
поставка     → привозит в точку
холодильник  → хранит (резерв)
витрина      → показывает (остатки)
заказ        → забирает порцию
```
