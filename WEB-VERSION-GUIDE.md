## 🎯 НАЗНАЧЕНИЕ

Этот документ описывает особенности работы с репозиторием VK-offee через веб-версию Claude (claude.ai) и как избежать типичных проблем.

---

## ⚠️ ГЛАВНАЯ ПРОБЛЕМА: Ограничения веток

### Проблема

Веб-версия Claude НЕ МОЖЕТ пушить в ветки, которые не начинаются с `claude/`.

**Ошибка:**
```
error: failed to push some refs to 'https://github.com/...'
HTTP 403: Forbidden
```

### Причина

Правила репозитория требуют, чтобы ветки веб-агентов начинались с `claude/` и заканчивались идентификатором сессии.

**Правильный формат:**
```
claude/название-задачи-<sessionID>
```

**Примеры:**
- ✅ `claude/fpf-integration-EIfbT`
- ✅ `claude/google-drive-sync-setup-EIfbT`
- ❌ `feature/fpf-srt-integration` (не начинается с claude/)
- ❌ `main` (защищённая ветка)

## ✅ РЕШЕНИЕ: Как работать правильно

### Вариант 1: Работать в правильной ветке (РЕКОМЕНДУЕТСЯ)

**Перед началом работы:**

1. Проверь текущую ветку:
   ```bash
   git branch --show-current
   ```

2. Если ветка НЕ начинается с `claude/`, создай новую:
   ```bash
   git checkout -b claude/название-задачи-<sessionID>
   ```

3. Работай в этой ветке:
   ```bash
   git add .
   git commit -m "описание"
   git push -u origin claude/название-задачи-<sessionID>
   ```

### Вариант 2: Работать локально (если нет прав)

**Если не можешь создать ветку:**

1. Работай локально:
   ```bash
   git add .
   git commit -m "описание"
   ```

2. Сообщи пользователю:
   ```
   ✅ Изменения сохранены локально (коммит XXXXXX)
   ⚠️ Не могу запушить в ветку feature/название

   Причина: Ветка должна начинаться с claude/

   Решение: CLI-версия Claude запушит изменения
   ```

3. CLI-версия заберёт коммиты и запушит

## 🔧 ТИПИЧНЫЕ ПРОБЛЕМЫ И РЕШЕНИЯ

### Проблема 1: Ошибка 403 при push

**Симптомы:**
```
error: failed to push some refs
HTTP 403: Forbidden
```

**Причина:** Ветка не начинается с `claude/`

**Решение:**
1. Проверь ветку: `git branch --show-current`
2. Если не `claude/*`, создай новую: `git checkout -b claude/название-<sessionID>`
3. Перенеси коммиты: `git cherry-pick <hash>`
4. Запуши: `git push -u origin claude/название-<sessionID>`

---

### Проблема 2: Не знаю sessionID

**Симптомы:**
Нужно создать ветку `claude/название-<sessionID>`, но не знаю sessionID

**Решение:**
1. Используй любой уникальный идентификатор (5-10 символов)
2. Примеры: `EIfbT`, `Xa9Kp`, `Qw3Rt`
3. Или используй дату: `claude/название-20260221`

---

### Проблема 3: Хочу работать в feature/название

**Симптомы:**
Есть ветка `feature/fpf-srt-integration`, хочу в ней работать

**Решение:**
1. Работай локально (без push)
2. Коммить изменения
3. Сообщи пользователю:
   ```
   ✅ Коммит создан: XXXXXX
   ⚠️ Не могу запушить (ветка не начинается с claude/)

   CLI-версия Claude запу��ит изменения
   ```

---

### Проблема 4: Нужно мержить в main

**Симптомы:**
Хочу смержить изменения в `main`

**Решение:**
**НЕ ДЕЛАЙ ЭТОГО!** Веб-версия не может мержить в `main`.

1. Работай в своей ветке `claude/название-<sessionID>`
2. Запуши изменения
3. Сообщи пользователю:
   ```
   ✅ Изменения в ветке claude/название-<sessionID>

   Для мержа в main:
   1. CLI-версия проверит изменения
   2. Запустит тесты
   3. Смержит в main
   ```

## 📝 ПРАВИЛА КОММУНИКАЦИИ

### Выдавай информацию маленькими блоками

**Правило:**
- Максимум 10-15 строк текста за раз
- После блока — пауза для вопросов
- Не выдавай "простыни" текста

**Почему:**
- Пользователю сложно читать большие блоки
- Легче задавать вопросы по ходу
- Меньше вероятность недопонимания

**Пример (ПЛОХО):**
```
Вот полный отчёт на 200 строк...
[огромная простыня текста]
```

**Пример (ХОРОШО):**
```
Проверил репозиторий. Нашёл 3 проблемы:

1. .gitignore не содержит config/
2. CLAUDE.md не содержит правило о блоках
3. Ветка не начинается с claude/

Начать исправление? (Да/Нет)
```

## 📊 СТАТИСТИКА ПРОБЛЕМ

### Проблемы, с которыми сталкивались:

1. **Ошибка 403 при push** (100% случаев)
   - Причина: Ветка не начинается с `claude/`
   - Решение: Создать правильную ветку

2. **Смешивание веток** (50% случаев)
   - Причина: Попытка перенести коммиты на старую ветку
   - Решение: Работать в своей ветке

3. **Потеря коммитов** (20% случаев)
   - Причина: Не сообщили пользователю о локальных коммитах
   - Решение: Всегда сообщать хеш коммита
