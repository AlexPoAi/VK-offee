# CODEX.METHOD.001 — Приём задач из Telegram в Codex Runtime

## Назначение

Единая последовательность действий для безопасного intake задач из режима `🤖 Codex`
в Telegram-боте без поломки существующего RAG-контура.

## Предусловия

1. `RAG mode` и `Codex mode` разделены как разные маршруты внутри бота.
2. `Codex mode` доступен только авторизованному личному чату или явному allowlist.
3. Существует storage contract `session/task/artifact/outbox`.
4. Intake-контур не подменяет обычный вопрос в базу знаний.

## Каноническая последовательность

1. Пользователь явно входит в `Codex mode` через кнопку `🤖 Codex` или команду `/codex`.
2. Transport Bot проверяет, что сообщение пришло из разрешённого чата.
3. Бот определяет, к какой `session` относится входящее:
   - продолжить активную session;
   - или открыть новую session.
4. Создаётся `task-card` с типом входа:
   - `task`
   - `design`
   - `follow-up`
   - `artifact-only`
5. Все вложения сохраняются в `artifacts/` и привязываются к `task`.
6. В `registry/` обновляется mapping:
   - `telegram chat/thread/message -> session/task`.
7. Пользователь получает truthful-ответ о приёме:
   - задача принята;
   - номер session/task;
   - ожидание обработки.
8. Если deep-processing недоступен, intake всё равно завершается успешно как `accepted`,
   а не ломает маршрут и не уводит сообщение в RAG.

## Результат метода

После выполнения метода существуют:
- живая `session`;
- входящий `task-card`;
- привязанные `artifacts`, если были;
- запись в `registry`;
- подтверждение пользователю в Telegram.

## Критерий завершения

- входящее сообщение не потеряно;
- задача не ушла в RAG по ошибке;
- есть минимальный след в `session/task/outbox`;
- пользователь получил подтверждение приёма.

## Типовые ошибки

### E1. Codex mode смешан с RAG mode

Симптом: сообщение пользователя то идёт в базу знаний, то в intake.

Коррекция: переключение режима должно быть явным и устойчивым.

### E2. Нет session binding

Симптом: каждый новый месседж создаёт отдельную несвязанную задачу.

Коррекция: сначала определить/создать session, потом писать task.

### E3. Фото принято, но не связано с задачей

Симптом: файл сохранён, но непонятно, к какому ходу он относится.

Коррекция: артефакт должен иметь owner-task или owner-session.

### E4. Недоступен deep-processing, и бот молчит

Симптом: пользователь думает, что задача принята в работу, но это не так.

Коррекция: truthful ответ — принято в inbox, обработка позже.
