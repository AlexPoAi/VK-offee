## 🎯 Миссия репозитория

VK-offee - это операционная база знаний сети кофеен «Вкусный Кофе». Репозиторий содержит все процессы, стандарты, рецепты и инструменты для управления кофейнями.

**Цель:** Обеспечить единый источник истины (Single Source of Truth) для всех операционных процессов на базе FPF+SPF+SRT+FMT архитектуры.

## 🏛️ FPF: First Principles Framework

### Что это

**FPF** — "операционная система для мышления", трансдисциплинарная архитектура рассуждений.

### Ключевые принципы для VK-offee

| Принцип | Паттерн | Применение в VK-offee |
|---------|---------|----------------------|
| **Холонический фундамент** | A.1 | Кофейня — холон: часть сети и целое само по себе |
| **Ограниченный контекст** | A.1.1 | Термины определены в границах домена кофеен |
| **Строгое различение** | A.7 | Роль ≠ Метод ≠ Работа (см. distinctions.md) |
| **F-G-R доверие** | B.3 | Доверие = (Формальность, Область, Надёжность) |
| **Цикл ADI** | B.5 | Абдукция → Дедукция → Индукция |
| **Трансформеры** | A.7 | Agent→Method→Description→Work |

### Когда обращаться к FPF

**Обязательно:**
- При создании новых процессов (методов)
- При определении ролей и их границ
- При возникновении концептуальных проблем
- При рефакторинге архитектуры

**Как использовать:**
1. Открой `.fpf/FPF-Spec.md`
2. Найди релевантный паттерн (Ctrl+F)
3. Примени к своей задаче
4. Задокументируй решение

## 📋 SPF: Second Principles Framework

### Что это

**SPF** — методология создания Pack (формализованного доменного знания).

### 12 стадий процесса создания знаний

| Стадия | Файл | Что делаем |
|--------|------|------------|
| 00 | process-overview.md | Понимаем процесс |
| 01 | domain-selection.md | Выбираем домен |
| 02 | bounded-context.md | Определяем границы |
| 03 | distinctions-work.md | Создаём различения |
| 04 | domain-entities-identification.md | Идентифицируем сущности |
| 05 | information-ingestion.md | Принимаем информацию |
| 06 | analysis-and-formalization.md | Анализируем и формализуем |
| 07 | method-and-product-extraction.md | Извлекаем методы и продукты |
| 08 | failure-modes-extraction.md | Извлекаем типовые ошибки |
| 09 | sota-annotation.md | Присваиваем SoTA-статус |
| 10 | map-maintenance.md | Поддерживаем карты |
| 11 | review-and-evolution-cycle.md | Цикл пересмотра |

### Process Lint (ОБЯЗАТЕЛЬНО!)

**Перед каждым коммитом в PACK-cafe-operations:**

```bash
# Проверь process-lint.md
# Убедись, что:
- [ ] Нет дидактического языка (step, lesson, try this)
- [ ] Method не является сценарием
- [ ] Work Product имеет критерии существования
- [ ] Failure Mode имеет тест обнаружения
- [ ] Структурные изменения отражены в map
- [ ] SoTA имеет критерий пересмотра
```

### Hard Gates (блокируют коммит)

| Gate | Условие | Решение |
|------|---------|---------|
| HG-1 | Method без WP link | Добавить ссылку на Work Product |
| HG-2 | Method без distinction | Добавить ссылку на различение |
| HG-3 | WP без existence criteria | Добавить критерии |
| HG-4 | FM без detection test | Добавить тест |
| HG-5 | Structural change без map update | Обновить карту |
| HG-6 | Didactic language | Переписать |
| HG-7 | Scenario вместо method | Переписать |
| HG-8 | SoTA без revision criterion | Добавить критерий |

## 🗂️ SRT: Systems-Roles-Table

### Таблица 3×3

```
                     Предприниматель  Инженер           Менеджер
                     (Зачем?)         (Как устроено?)   (Как работает?)
┌────────────────────────────────────────────────────────────────────┐
│ Suprasystem        F1               F2                F3           │
│ (Надсистема)       Контекст         Окружение         Взаимодействие│
├────────────────────────────────────────────────────────────────────┤
│ System-of-Interest F4               F5                F6           │
│ (Целевая система)  Требования       Архитектура       Реализация   │
├────────────────────────────────────────────────────────────────────┤
│ Constructor        F7               F8                F9           │
│ (Система создания) Принципы         Платформа         Команда      │
└────────────────────────────────────────────────────────────────────┘
```

**F0** — метасемейство (управление хранилищем знаний)

### Три системы для VK-offee

| Система | Что это |
|---------|---------|
| **Suprasystem (F1-F3)** | Рынок HoReCa, клиенты кофеен, партнёры |
| **System-of-Interest (F4-F6)** | Сеть кофеен VK-offee (продукт) |
| **Constructor (F7-F9)** | Команда менеджмента + инструменты |

### Как использовать SRT

**При создании документа спроси себя:**
1. К какой системе относится? (Suprasystem/SoI/Constructor)
2. Какую роль описывает? (Предприниматель/Инженер/Менеджер)
3. Положи в соответствующую папку `SRT/FX-*/`

**Примеры:**
- Анализ рынка кофеен → `F1-Suprasystem-Entrepreneur/`
- Рецепт капучино → `F6-SoI-Manager/`
- Telegram-бот → `F8-Constructor-Engineer/`
- Методология работы команды → `F9-Constructor-Manager/`

## 🔧 FMT: Практические протоколы

### Open-Work-Close

**Open (Открытие работы):**
1. Создай ветку `feature/название-задачи`
2. Обнови MEMORY.md с контекстом задачи
3. Создай задачу в WeekPlan.md

**Work (Работа):**
1. Следуй process/ стадиям
2. Используй шаблоны из `PACK-cafe-operations/_template/`
3. Запускай process-lint перед коммитом

**Close (Закрытие работы):**
1. Обнови карты в `PACK-cafe-operations/07-map/`
2. Обнови MEMORY.md с результатами
3. Создай PR с описанием изменений

### Extraction Protocol (Извлечение знаний)

**Из knowledge-base в PACK:**
1. Найди информацию в `knowledge-base/`
2. Примени `process/06-analysis-and-formalization.md`
3. Создай карточку метода в `PACK-cafe-operations/03-methods/`
4. Обнови индекс в `02-domain-entities/02C-methods-index.md`
5. Обнови карту в `07-map/`

## 🤖 Правила для AI-агентов

### ОБЯЗАТЕЛЬНО делать

1. **Читать FPF перед созданием процессов**
   - Открой `.fpf/FPF-Spec.md`
   - Найди релевантный паттерн
   - Примени к задаче

2. **Следовать SPF процессам**
   - Определи текущую стадию (00-11)
   - Создай требуемые work products
   - Не пропускай стадии

3. **Использовать SRT для организации**
   - Определи систему (Suprasystem/SoI/Constructor)
   - Определи роль (Предприниматель/Инженер/Менеджер)
   - Положи в правильную папку F0-F9

4. **Запускать process-lint**
   - Перед каждым коммитом в PACK
   - Проверь все Hard Gates
   - Исправь ошибки

5. **Обновлять карты**
   - После структурных изменений
   - В `PACK-cafe-operations/07-map/`
   - Связи между методами, WP, FM

6. **Выдавать информацию маленькими блоками**
   - Максимум 10-15 строк текста за раз
   - После блока — пауза для вопросов
   - Не выдавать "простыни" текста
   - Особенно важно для веб-версии Claude

### ЗАПРЕЩЕНО делать

1. **Дидактический язык**
   - ❌ "step", "lesson", "try this", "in N days"
   - ✅ Описывай что существует, не как учить

2. **Сценарии вместо методов**
   - ❌ "Сначала сделай X, потом Y"
   - ✅ "Метод X требует входы A, B и производит выход C"

3. **Смешивание понятий**
   - ❌ Роль = Метод = Работа
   - ✅ Строго различай (см. distinctions.md)

4. **Изменения без process-lint**
   - ❌ Коммит без проверки
   - ✅ Всегда проверяй перед коммитом

5. **Embeddings как source-of-truth**
   - ❌ Хранить только векторы
   - ✅ Markdown — единственный source-of-truth

## 📚 Связь с творческим конвейером

**Творческий конвейер** (`~/Documents/creativ-convector.nocloud/`) — Downstream/Governance система для обработки заметок из Obsidian.

### Иерархия принципов

```
Нулевые принципы (ZP) — работают везде
    ↓
Первые принципы (FPF) — работают в любой дисциплине
    ↓
Вторые принципы (Pack) — работают в домене → VK-offee
    ↓
Третьи принципы (Downstream) — работают в проекте → Творческий конвейер
```

### Протокол извлечения знаний

**Из Obsidian в VK-offee:**
1. Заметка попадает в `creativ-convector/1. Исчезающие заметки/`
2. AI-агент анализирует через призму FPF/SRT
3. Если относится к VK-offee → извлекается в PACK
4. Применяется `process/07-method-and-product-extraction.md`
5. Создаётся карточка метода в `PACK-cafe-operations/03-methods/`

## 🎯 Примеры использования

### Пример 1: Создание нового рецепта

```markdown
1. Определи стадию: 07-method-and-product-extraction
2. Используй шаблон: PACK-cafe-operations/_template/03-methods/
3. Создай файл: CAFE.METHOD.003-flat-white.md
4. Заполни секции:
   - ID: CAFE.METHOD.003
   - Name: Приготовление Flat White
   - Inputs: Эспрессо (30мл), Молоко (150мл)
   - Outputs: Flat White (180мл)
   - Distinctions: Меню ≠ Рецепт
   - Work Products: CAFE.WP.003-flat-white-checklist
5. Обнови индекс: 02-domain-entities/02C-methods-index.md
6. Обнови карту: 07-map/CAFE.MAP.001.md
7. Запусти process-lint
8. Коммит
```

### Пример 2: Добавление документа в SRT

```markdown
1. Документ: "Анализ конкурентов в районе"
2. Система: Suprasystem (окружение кофейни)
3. Роль: Предприниматель (зачем? контекст рынка)
4. Семейство: F1-Suprasystem-Entrepreneur
5. Путь: SRT/F1-Suprasystem-Entrepreneur/competitors-analysis-2026.md
6. Коммит
```

### Пример 3: Исправление ошибки process-lint

```markdown
❌ Ошибка: HG-1 (Method без WP link)

Файл: CAFE.METHOD.002-cappuccino.md

Исправление:
1. Открой файл
2. Найди секцию "Work Products"
3. Добавь: "- CAFE.WP.002-cappuccino-checklist"
4. Сохрани
5. Запусти process-lint снова
6. ✅ Проверка пройдена
```
